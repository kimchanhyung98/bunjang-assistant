# AI 작업 지침

```text
사용자가 지정한 상품 루트/
  {product}/
    **       # 사용자가 추가한 파일 전체
    note.md  # 선택: 상품별 메모와 입력값

스킬 내부 리소스/
  skills/bunjang/references/ai-context.md
  skills/bunjang/references/template.md
```

## 1. 기준 파일 확인 및 상품 디렉토리 선택

수행:
- 사용자가 지정한 상품 루트 또는 상품 디렉토리 경로를 기준으로 작업한다.
- `skills/bunjang/references/ai-context.md`는 현재 AI 작업 지침으로 사용한다.
- `skills/bunjang/references/template.md`는 판매글 템플릿으로 사용한다.
- 사용자가 “A 디렉토리의 상품들”이라고 요청하면 A를 상품 루트로 본다.
- 사용자가 지정한 상품 루트 바로 아래에 있는 디렉토리를 상품 디렉토리로 선택한다.
- 하위 디렉토리가 없고 지정 경로 자체에 상품 사진 또는 `note.md`가 있으면 지정 경로 자체를 단일 상품 디렉토리로 본다.
- 사용자가 지정한 상품 루트 바로 아래의 일반 파일은 여러 상품 처리 대상에서는 제외한다.
- 상품 디렉토리가 여러 개이면 하나씩 처리한다.
- 현재 상품 디렉토리 하나를 선택한다.

예외 처리:
- 디렉토리 이름만으로 상품을 알 수 없으면 디렉토리 이름을 임시 식별자로 사용한다.

중단 조건:
- 상품 디렉토리나 단일 상품으로 볼 수 있는 파일 구성이 하나도 없으면 작업을 중단한다.

주의 사항:
- 상품 디렉토리 이름이 상품명이라고 단정하지 않는다. `balenciaga-3xl` 같은 이름일 수도 있고 `001` 같은 번호일 수도 있다.

완료 기준:
- 현재 처리할 상품 디렉토리 경로가 하나로 확정되어 있다.
- 상품 디렉토리 이름을 상품명으로 쓸지 임시 식별자로만 쓸지 구분되어 있다.

## 2. 상품 디렉토리 파일 확인

수행:
- 현재 상품 디렉토리 안의 모든 하위 파일을 확인한다.
- 파일명과 확장자는 힌트로만 기록한다.
- 가능한 경우 파일 시그니처, MIME 타입, 이미지 메타데이터, 실제 디코딩 가능 여부를 확인한다.
- `note.md`가 있으면 상품명, SKU, 사이즈, 하자, 구성품, 희망 가격 같은 사용자 메모로 먼저 읽는다.

예외 처리:
- 읽을 수 없는 파일이 있으면 파일명을 기록하고 나머지 파일 확인을 계속한다.
- `note.md`가 없으면 이미지와 상품 식별에 필요한 공개 정보만으로 진행한다.

중단 조건:
- 현재 상품 디렉토리 안에 확인할 파일이 없으면 현재 상품 처리를 중단한다.

주의 사항:
- 특정 이미지 파일명이 있다고 가정하지 않는다.
- 사용자가 `img-01.jpg`로 올릴 수도 있고, `asdf.heic`처럼 임의 이름으로 올릴 수도 있다.
- 파일명이나 확장자만 보고 사용자 촬영 사진, 대표 이미지, 정제본 여부를 확정하지 않는다.

완료 기준:
- 현재 상품 디렉토리의 전체 파일 목록이 확보되어 있다.
- 읽을 수 없는 파일 목록이 있으면 파일명과 사유가 기록되어 있다.
- `note.md` 사용 여부와 메모 내용이 기록되어 있다.

## 3. 로컬 파일 이미지 판별 및 역할 분류

수행:
- 각 파일을 실제로 열어 이미지로 디코딩 가능한지 확인한다.
- 디코딩 가능한 이미지의 실제 포맷과 업로드 가능 여부를 확인한다.
- JPEG 또는 PNG로 확인된 이미지는 업로드 가능한 이미지 후보로 분류한다.
- JPEG/PNG가 아니지만 디코딩 가능한 이미지는 분석 가능한 이미지 후보로 분류하고 업로드용 정제 대상으로 표시한다.
- `official-*`로 시작하는 디코딩 가능한 이미지는 기존 대표 이미지 후보로 분류한다.
- `upload-*`로 시작하는 디코딩 가능한 이미지는 기존 정제본 후보로 분류한다.
- `official-*`, `upload-*`가 아닌 디코딩 가능한 이미지는 사용자 촬영 사진 후보로 분류한다.
- 디코딩 가능한 이미지가 아닌 파일은 상품 이미지 후보에서 제외한다.

예외 처리:
- `official-01.jpg`뿐 아니라 `official-1.heic`, `official-front.png` 같은 파일도 기존 대표 이미지 후보로 본다.
- 기존 대표 이미지 후보가 JPEG/PNG가 아니지만 디코딩 가능하면 대표 이미지 정제 대상으로 표시한다.
- 확장자가 없거나 MIME 타입이 불명확한 파일은 열어볼 수 있을 때만 이미지 후보로 유지한다.
- 알려진 이미지 확장자여도 열 수 없으면 이미지 후보에서 제외하고 파일명을 기록한다.
- 확장자와 실제 포맷이 다르면 실제로 디코딩된 포맷을 기준으로 판단한다.
- 동일한 이미지가 여러 이름으로 있으면 하나만 업로드 후보로 남긴다.

중단 조건:
- 사용자 촬영 사진 후보가 0장이면 현재 상품 처리를 중단한다.

주의 사항:
- `.webp`, `.heic`, `.heif`, `.avif`는 변환이 필요할 수 있는 예시일 뿐 전체 목록이 아니다.
- `official-*` 파일은 사용자가 직접 추가한 참고 이미지일 수 있다.
- `official-*` 파일은 사용자 촬영 사진으로 취급하지 않는다.
- `upload-*` 파일은 정제본 후보이며, 원본 사용자 촬영 사진으로 취급하지 않는다.
- 번개장터 파일 입력의 허용 형식은 `accept="image/jpeg, image/png"`로 간주한다.
- 최종 업로드 후보는 JPEG 또는 PNG여야 한다.

완료 기준:
- 사용자 촬영 사진 후보, 기존 대표 이미지 후보, 기존 정제본 후보가 분리되어 있다.
- 각 이미지 후보의 실제 포맷과 업로드 가능 여부가 기록되어 있다.
- 이미지 후보에서 제외한 파일이 있으면 파일명과 제외 사유가 기록되어 있다.

## 4. 사용자 촬영 사진 검증 및 정제

수행:
- 사용자 촬영 사진 후보를 열어 파일 손상 여부를 확인한다.
- EXIF 회전, 중복 이미지, 심한 흔들림, 상품 식별 불가 여부를 확인한다.
- 업로드 가능한 원본에 문제가 없으면 그대로 사용한다.
- 업로드 전 변환이나 보정이 필요한 파일은 같은 상품 디렉토리에 `upload-01.jpg`, `upload-02.jpg` 같은 JPEG 정제본을 만든다.

예외 처리:
- 일부 이미지가 손상됐으면 해당 파일을 제외하고 남은 이미지로 진행 가능 여부를 판단한다.
- 원본은 정상이나 업로드 형식이 아니면 JPEG 정제본을 만든다.
- 원본과 정제본이 같은 내용을 담고 있으면 정제본을 업로드 후보로 사용한다.

중단 조건:
- 이미지 파일을 열 수 없고 대체 이미지도 없으면 현재 상품 처리를 중단한다.
- 업로드용 정제본이 필요한데 변환에 실패하면 현재 상품 처리를 중단한다.
- 남은 이미지로 상품을 식별할 수 없으면 현재 상품 처리를 중단한다.
- 서로 다른 상품 사진이 섞여 있으면 현재 상품의 자동입력을 중단한다.

주의 사항:
- 원본 파일은 삭제하지 않는다.
- 원본 파일은 덮어쓰지 않는다.
- 정제본은 같은 상품 디렉토리에 만든다.
- 업로드 목록에는 JPEG 또는 PNG가 아닌 파일을 포함하지 않는다.

완료 기준:
- 업로드 가능한 사용자 촬영 사진 원본 또는 `upload-*` 정제본 목록이 확정되어 있다.
- 생성한 정제본이 있으면 원본 파일과 정제본 파일의 대응 관계가 기록되어 있다.
- 제외한 사용자 촬영 사진이 있으면 파일명과 제외 사유가 기록되어 있다.

## 5. 상품 정보 추론 및 판매글 작성

수행:
- `skills/bunjang/references/template.md`를 읽는다.
- 상품 사진, `note.md`, 박스 라벨, 로고, SKU를 기준으로 상품 유형, 브랜드/모델명, 사이즈, 색상, 상태, 하자를 추론한다.
- 템플릿 형식에 맞춰 판매글 초안을 작성한다.
- 신발 판매글은 브랜드, 모델명, 사이즈, 상태, 오염/하자, 구성품 정보를 반드시 확인한다.
- 특이사항은 20자 내외로 간단하게 작성한다.

예외 처리:
- 이미지 인식 결과와 라벨, SKU, 공개 상품 정보가 다르면 더 강한 근거를 우선한다.
- 확실하지 않은 정보는 판매글에 단정적으로 쓰지 않는다.
- 하자 여부가 애매하면 사진 근거와 함께 보수적으로 표현한다.

중단 조건:
- 상품 유형이나 상품명을 전혀 추론할 수 없으면 현재 상품 처리를 중단한다.
- 판매글에 필요한 핵심 정보가 부족하고 사진에서도 보완할 수 없으면 현재 상품 처리를 중단한다.

주의 사항:
- 하자, 오염, 사용감은 숨기지 않는다.
- 사용자가 제공하지 않은 구성품을 포함한다고 쓰지 않는다.
- 템플릿의 빈 항목을 임의 사실로 채우지 않는다.
- 신발/의류는 사이즈와 카테고리를 확인하고, 사이즈가 없는 상품군은 사이즈 필드를 임의로 만들지 않는다.

완료 기준:
- 판매글 초안이 작성되어 있다.
- 추론한 정보와 사용자가 제공한 정보가 구분되어 있다.
- 불확실한 정보가 있으면 단정하지 않은 형태로 표시되어 있다.

## 6. 대표 이미지 확인

수행:
- 로컬 이미지 역할 분류에서 확인한 기존 대표 이미지 후보를 먼저 확인한다.
- 기존 대표 이미지 후보가 상품 단독 이미지이고 JPEG/PNG이면 최종 대표 이미지로 사용한다.
- 기존 대표 이미지 후보가 상품 단독 이미지지만 JPEG/PNG가 아니면 같은 상품 디렉토리에 `official-01.jpg`, `official-02.jpg` 같은 JPEG 정제본을 만든다.
- 사용할 수 있는 기존 대표 이미지가 없으면 웹검색 이미지는 상품 식별 참고용으로만 확인한다.
- 웹검색 이미지를 다운로드하거나 업로드 후보로 사용하지 않는다.
- 최종 대표 이미지는 최대 2장까지 사용한다.

예외 처리:
- 기존 대표 이미지 후보가 상품과 맞지 않으면 제외하고 파일명을 기록한다.
- 기존 대표 이미지 후보를 열 수 없으면 제외하고 파일명을 기록한다.
- 기존 대표 이미지 후보 변환에 실패하면 제외하고 파일명을 기록한다.
- 기존 대표 이미지가 2장을 넘으면 상품 단독 식별이 가장 쉬운 이미지 2장을 우선한다.
- 저장할 파일명이 이미 있으면 다음 사용 가능한 번호를 사용한다.
- 기존 대표 이미지 후보가 없으면 `대표 이미지 없음`으로 기록하고 사용자 촬영 사진만으로 진행한다.
- 대표 이미지가 없더라도 사용자 촬영 사진으로 상품 식별, 상태 설명, 업로드가 가능하면 현재 상품 처리를 계속한다.

중단 조건:
- 대표 이미지와 사용자 촬영 사진을 포함해 업로드 가능한 JPEG 또는 PNG 이미지가 전혀 없으면 현재 상품 처리를 중단한다.
- 대표 이미지를 확보했지만 JPEG 또는 PNG 업로드 후보를 만들 수 없고 사용자 촬영 사진만으로도 진행할 수 없으면 현재 상품 처리를 중단한다.

주의 사항:
- 대표 이미지는 사용자가 직접 추가했을 수도 있다.
- 웹검색 이미지는 식별 참고용이며 다운로드하거나 업로드하지 않는다.
- 대표 이미지는 상품 단독 이미지여야 한다.
- 실제 상품 상태를 왜곡하는 대표 이미지는 사용하지 않는다.
- 원본 대표 이미지 파일은 삭제하거나 덮어쓰지 않는다.
- 대표 이미지를 확보하지 못한 것만으로는 현재 상품 처리를 중단하지 않는다.

완료 기준:
- 최종 대표 이미지가 최대 2장으로 확정되어 있거나 `대표 이미지 없음`으로 기록되어 있다.
- 각 대표 이미지의 출처가 로컬 파일 또는 로컬 변환 중 무엇인지 기록되어 있다.
- 사용할 수 없는 대표 이미지 후보가 있으면 파일명과 제외 사유가 기록되어 있다.

## 7. 시세 조사

수행:
- 시세 조회는 `skills/bunjang/references/price.md`를 단일 기준으로 따른다. 즉 `search.listings`/`agent-search-rank`로 번개장터 내 동일/유사 매물 가격을 먼저 확인한다.
- 번개장터 내 결과가 부족할 때만 웹검색을 보조 근거로 사용한다.
- `note.md`에 희망 가격이 있으면 가장 먼저 확인하고, `price.md`가 제시하는 범위와 충돌하면 양쪽을 모두 보고한다.
- 등록가는 `price.md`가 산정한 시세 범위 안에서 사용자의 희망 가격, 상품 상태, 구성품을 반영해 결정한다. "평균 +N%" 같은 단정적인 가산 규칙을 임의로 적용하지 않는다.
- 가격 근거를 판매글 초안과 별도로 기록한다.

예외 처리:
- 번개장터 검색 결과가 부족하면 키워드를 변형해 재검색한다. 그래도 부족하면 웹검색을 보조로 쓴다.
- 동일 상품이 없으면 같은 브랜드, 같은 카테고리, 비슷한 상태의 유사 상품 시세를 참고한다.

중단 조건:
- 번개장터 검색과 웹검색 모두에서 가격 근거를 전혀 얻지 못하면 가격 근거 부족으로 보고하고 현재 상품 처리를 중단한다.

주의 사항:
- 지나치게 낮거나 높은 이상치 가격을 기준으로 삼지 않는다.
- 상태, 구성품, 사이즈 차이를 가격 판단에 반영한다.
- 웹검색 결과를 판매글에 과장해서 쓰지 않는다.
- 추천 등록가는 확정 가격이 아니라 등록 시작가다.

완료 기준:
- 등록가가 숫자로 확정되어 있다.
- 가격 근거가 된 검색어와 참고 시세가 기록되어 있다.
- 번개장터 내 시세가 부족해 웹검색을 보조로 썼다면 그 사실이 기록되어 있다.

## 8. 업로드 목록 생성

수행:
- 최종 대표 이미지가 있으면 먼저 둔다.
- 사용자 촬영 사진 원본 또는 `upload-*` 정제본을 이어서 둔다.
- 전체 업로드 이미지는 최대 12장으로 제한한다.
- 대표 이미지는 최대 2장까지 사용한다.
- 대표 이미지를 제외한 사용자 촬영 사진은 최대 10장까지 사용한다.

예외 처리:
- 원본과 정제본이 같은 내용을 담고 있으면 정제본을 우선한다.
- 이미지 수가 12장을 넘으면 대표 이미지, 전체 외관, 라벨/SKU, 하자 사진 순서로 우선한다.

중단 조건:
- 최종 업로드 목록에 넣을 JPEG 또는 PNG 파일이 없으면 현재 상품 처리를 중단한다.

주의 사항:
- 업로드 목록에는 JPEG 또는 PNG가 아닌 파일을 포함하지 않는다.
- 같은 이미지를 중복으로 넣지 않는다.
- 실제 이미지 파일은 기본적으로 git에 커밋하지 않는다.

완료 기준:
- 최종 업로드 이미지 순서가 확정되어 있다.
- 업로드 목록의 모든 파일이 JPEG 또는 PNG로 확인되어 있다.
- 제외한 이미지가 있으면 파일명과 제외 사유가 기록되어 있다.

## 9. 명시 요청 시 번개장터 입력

수행:
- 사용자가 브라우저 입력을 명시한 경우에만 판매글을 하나씩 처리한다.
- 최종 업로드 목록 순서대로 이미지를 업로드한다.
- 제목, 가격, 설명, 카테고리, 상품 상태, 배송 정보, 태그를 입력한다.
- 태그는 하나씩 입력하고 스페이스바로 확정한다.

예외 처리:
- 이미지 업로드가 실패하면 같은 파일로 한 번 재시도한다.
- 태그 입력이 확정되지 않으면 태그 값을 다시 입력하고 스페이스바로 확정한다.
- 페이지 입력값이 사라지면 현재 상품의 초안과 업로드 목록 기준으로 다시 입력한다.

중단 조건:
- 이미지 업로드가 재시도 후에도 실패하면 현재 상품 입력을 중단한다.
- 필수 입력값을 채울 수 없으면 현재 상품 입력을 중단한다.

주의 사항:
- 이미지 중복 업로드 여부를 확인한다.
- `등록하기`는 절대 누르지 않는다.
- `임시 저장`도 누르지 않는다.

완료 기준:
- 번개장터 화면에 입력한 필드와 입력하지 못한 필드가 구분되어 있다.
- 업로드한 이미지 수와 태그 입력 상태가 확인되어 있다.
- `등록하기`와 `임시 저장`을 누르지 않은 상태가 확인되어 있다.

## 10. 결과 보고 및 다음 상품 처리

수행:
- 사용한 대표 이미지 파일명을 보고하고, 없으면 `대표 이미지 없음`으로 보고한다.
- 사용한 사용자 촬영 사진 또는 정제본 파일명을 보고한다.
- 추론한 상품명, 가격, 주요 하자, 남은 주의사항을 보고한다.
- 다음 상품 디렉토리가 있으면 현재 상품 기록을 닫고 다음 상품을 별도 작업 단위로 시작한다.

예외 처리:
- 현재 상품이 중단됐으면 중단된 단계와 원인을 보고한다.
- 일부 파일을 제외하고 진행했으면 제외한 파일명과 이유를 보고한다.

중단 조건:
- 보고할 결과나 실패 원인을 정리할 수 없으면 작업을 멈춘다.

주의 사항:
- 실패한 상품이 있어도 다른 상품 디렉토리까지 자동으로 건너뛰지 않는다.
- 다음 상품으로 넘어가기 전 현재 상품의 상태를 명확히 남긴다.

완료 기준:
- 현재 상품의 처리 상태가 완료, 중단, 확인 필요 중 하나로 정리되어 있다.
- 처리한 파일, 생성한 파일, 제외한 파일이 구분되어 있다.
- 남은 상품 디렉토리 여부가 기록되어 있다.
